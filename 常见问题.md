## 基础概念

### Q1: 什么是线程安全？

**答**：线程安全是指当多个线程访问某个类时，这个类始终能表现出正确的行为。具体来说：

- **原子性**：操作要么全部完成，要么都不完成
- **可见性**：一个线程的修改对其他线程可见
- **有序性**：指令执行顺序符合预期

### Q2: volatile 能保证线程安全吗？

**答**：不能完全保证。

- ✅ volatile 可以保证：
  - 可见性：写操作对其他线程立即可见
  - 有序性：禁止指令重排序

- ❌ volatile 不能保证：
  - 复合操作的原子性：`counter++` 不是原子的

**正确使用场景**：
- 状态标记：`volatile boolean stopRequested = false;`
- 单例模式：double-checked locking 中的变量声明
- 简单读写：一个线程写，多个线程读

### Q3: synchronized 和 Lock 有什么区别？

| 特性 | synchronized | Lock |
|------|-------------|------|
| 使用方式 | 关键字 | 接口 |
| 锁获取 | 自动 | 手动 |
| 锁释放 | 自动（代码块结束） | 手动（finally） |
| 可中断 | 否 | 是（lockInterruptibly） |
| 超时 | 否 | 是（tryLock） |
| 公平性 | 非公平 | 可配置 |
| 条件变量 | wait/notify | Condition |
| 性能 | JDK 6+ 优化后很好 | 相当 |

**推荐**：
- 简单场景使用 synchronized
- 复杂场景使用 Lock（需要可中断、超时、公平性等）

### Q4: 什么是线程饥饿？

**答**：线程饥饿是指某个线程长时间无法获得需要的资源而无法执行。

**常见原因**：
- 优先级过低的线程一直得不到 CPU
- 低优先级线程被高优先级线程"饿死"
- 锁竞争激烈，某个线程一直获取不到锁

**解决方案**：
- 使用公平锁
- 避免长时间持有锁
- 设置合理的线程优先级

## 锁机制

### Q5: 什么是死锁？

**答**：死锁是指两个或多个线程互相等待对方持有的锁，导致所有线程都无法继续执行。

**死锁的四个必要条件**：
1. **互斥条件**：资源不能共享
2. **持有并等待**：持有资源同时等待其他资源
3. **不可剥夺**：资源不能被强制剥夺
4. **循环等待**：形成等待环路

**预防措施**：
- 统一锁获取顺序
- 使用 tryLock 带超时
- 减少锁持有时间
- 检测死锁（ThreadMXBean）

### Q6: 什么是活锁？

**答**：活锁是指线程一直在运行，但无法推进工作。

**典型场景**：
```java
// 两个线程互相让步
while (true) {
    if (lock1.tryLock()) {
        try {
            if (lock2.tryLock()) {
                // 完成工作
                break;
            }
        } finally {
            lock1.unlock();
        }
    }
    // 退避，但可能一直重试
}
```

**解决方案**：
- 引入随机退避
- 设置重试上限
- 使用固定的锁顺序

### Q7: StampedLock 是什么？

**答**：StampedLock 是 Java 8 引入的锁实现，支持三种模式：

1. **乐观读**：不阻塞写操作，性能最好
2. **悲观读**：共享读锁，类似 ReadWriteLock
3. **写锁**：排他锁

**使用场景**：
- 读多写少
- 读操作非常快
- 对读性能要求高

**注意**：
- 不可重入
- 没有 Condition 支持

## 并发集合

### Q8: ConcurrentHashMap 的原理是什么？

**答**：ConcurrentHashMap 使用分段锁机制（JDK 7）或 CAS + synchronized（JDK 8）：

**JDK 8 实现**：
- 使用 Node 数组 + 链表/红黑树
- 读操作无锁（volatile）
- 写操作使用 CAS + synchronized
- 支持高并发

### Q9: CopyOnWriteArrayList 适用什么场景？

**答**：CopyOnWriteArrayList 写时复制，适用场景：

**适用**：
- 读多写少
- 遍历期间数据不变
- 事件监听器列表

**不适用**：
- 频繁写入
- 大量数据
- 内存敏感场景

### Q10: BlockingQueue 有哪些实现？

**答**：主要实现及特点：

| 实现 | 特点 | 适用场景 |
|------|------|----------|
| ArrayBlockingQueue | 有界，数组 | 固定容量，需要限流 |
| LinkedBlockingQueue | 可选有界，链表 | 生产者消费者 |
| PriorityBlockingQueue | 优先级，无界 | 有序任务 |
| SynchronousQueue | 零容量 | 手递手传递 |
| DelayQueue | 延迟取出 | 定时任务 |

## 线程池

### Q11: 线程池的核心参数有哪些？

**答**：ThreadPoolExecutor 的核心参数：

```java
new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数
    maximumPoolSize,   // 最大线程数
    keepAliveTime,     // 空闲线程存活时间
    TimeUnit.SECONDS,  // 时间单位
    workQueue,         // 任务队列
    threadFactory,     // 线程工厂
    handler            // 拒绝策略
)
```

**参数说明**：
- `corePoolSize`：核心线程，即使空闲也不会回收
- `maximumPoolSize`：最大线程数
- `workQueue`：任务队列，存放等待执行的任务
- `handler`：拒绝策略，队列满时的处理方式

### Q12: 为什么不建议使用 Executors 工厂方法？

**答**：Executors 工厂方法存在以下问题：

1. **FixedThreadPool / SingleThreadExecutor**
   - 使用无界 LinkedBlockingQueue
   - 可能导致 OOM

2. **CachedThreadPool**
   - 创建无限线程
   - 可能导致资源耗尽

3. **ScheduledThreadPool**
   - 使用无界队列
   - 任务堆积问题

**推荐**：
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,
    maximumPoolSize,
    keepAliveTime,
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100), // 有界队列
    new ThreadFactoryBuilder().setNameFormat("worker-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 明确的拒绝策略
);
```

### Q13: 如何设置线程池大小？

**答**：根据任务类型设置：

**CPU 密集型**：
```java
int corePoolSize = Runtime.getRuntime().availableProcessors() + 1;
```

**IO 密集型**：
```java
int corePoolSize = Runtime.getRuntime().availableProcessors() * 2;
```

**混合型**：
- 根据实际情况调整
- 监控线程池状态
- 动态调整参数

## 异步编程

### Q14: CompletableFuture 和 Future 有什么区别？

**答**：CompletableFuture 相比 Future 提供了更强大的功能：

| 特性 | Future | CompletableFuture |
|------|--------|-------------------|
| 链式调用 | 不支持 | 支持（thenApply, thenAccept 等） |
| 组合 | 不支持 | 支持（thenCombine, allOf 等） |
| 异常处理 | get() 抛出 | exceptionally, handle |
| 手动完成 | 不支持 | complete() |
| 回调 | 不支持 | 支持 |

### Q15: 如何处理 CompletableFuture 的异常？

**答**：三种方式：

1. **exceptionally**：恢复异常，返回默认值
   ```java
   future.exceptionally(e -> {
       log.error("处理失败", e);
       return defaultValue;
   });
   ```

2. **handle**：无论成功失败都处理
   ```java
   future.handle((result, e) -> {
       if (e != null) {
           return errorValue;
       }
       return result;
   });
   ```

3. **whenComplete**：只用于副作用
   ```java
   future.whenComplete((result, e) -> {
       if (e != null) {
           log.error("处理失败", e);
       }
   });
   ```

## 性能问题

### Q16: 如何优化并发性能？

**答**：主要优化方向：

1. **减少锁竞争**
   - 缩小锁范围
   - 使用读写锁
   - 使用乐观锁（CAS）

2. **减少上下文切换**
   - 合理设置线程数
   - 使用协程/虚拟线程
   - 减少锁等待时间

3. **减少对象创建**
   - 使用对象池
   - 复用对象

4. **使用高效的数据结构**
   - ConcurrentHashMap vs HashMap
   - LongAdder vs AtomicLong

### Q17: 什么是伪共享（False Sharing）？

**答**：伪共享是指多个线程访问同一缓存行的不同变量，导致缓存行失效，影响性能。

**示例**：
```java
class Counter {
    volatile long count1;
    volatile long count2; // 可能与 count1 在同一缓存行
}
```

**解决方案**：
```java
@sun.misc.Contended  // JDK 8+
class Counter {
    volatile long count1;
    @sun.misc.Contended
    volatile long count2;
}
```

或者手动填充：
```java
class Counter {
    volatile long count1;
    long p1, p2, p3, p4, p5, p6, p7; // 填充
    volatile long count2;
}
```

### Q18: 如何检测并发问题？

**答**：主要方法：

1. **静态分析**
   - FindBugs
   - SpotBugs
   - SonarQube

2. **动态检测**
   - ThreadMXBean.findDeadlockedThreads()
   - JFR（Java Flight Recorder）

3. **单元测试**
   - 多次运行测试
   - 使用多线程测试
   - 并发压力测试

4. **代码审查**
   - 重点审查锁的使用
   - 检查共享变量的访问
